# 代码审查

审查当前变更，重点找出高风险问题：功能回归、潜在 bug、架构退化、测试缺失。

## 审查范围

- 仅审查本次变更相关文件
- 包含未提交、已暂存、以及当前分支相对基线分支的差异
- 不扩展到无关模块

## 审查步骤

1. 确认变更文件
   - 使用 Git 查看本次改动清单
   - 先识别核心改动，再看关联文件

2. 逐文件检查
   - 理解改动目的与影响范围
   - 检查依赖关系和模块边界是否被破坏
   - 识别重复逻辑和不必要复杂度

3. 风险检查（必须）
   - 错误处理是否完整（Rust `Result` / TypeScript 异常路径）
   - 空值与边界条件是否覆盖
   - 异步流程是否有竞态或资源泄漏风险
   - React effect 是否有清理
   - Rust 中是否存在高风险用法（如不必要 `unwrap()`、async 场景误用阻塞）

4. 架构检查（重点）
   - 前后端边界是否清晰（前端通过 Tauri commands 访问后端能力）
   - UI、业务逻辑、数据访问是否混杂
   - 模块是否高内聚、低耦合

5. 测试检查（必须）
   - 新增或变更逻辑是否有对应测试
   - 错误路径和关键边界是否覆盖
   - 测试是否隔离、断言是否清晰

## 优先级

- **关键**：可能导致崩溃、数据丢失、安全问题、明显功能损坏
- **高**：高概率 bug、严重耦合或复杂度问题、关键逻辑缺少测试
- **中**：可维护性问题、可改进但不阻塞合并

## 输出格式

按下面结构输出，先问题后总结：

```markdown
## Findings
1. [关键/高/中] `path/to/file` - 问题标题
   - 影响：...
   - 建议：...

## Testing
- 已覆盖：...
- 缺失：...

## Summary
- 总体结论与风险
```

## 审查原则

- 结论必须基于本次改动证据
- 建议要具体、可执行
- 优先简单直接的修复方案
- 若发现关键或高优先级问题，审查结束时主动询问是否需要我直接修复
