# 发布流程

执行发布时，先保证质量，再执行发布。不要跳过问题，不要绕过检查。

## 目标

- 修复所有错误和警告后再发布
- 只提交和发布相关改动
- 让发布记录清晰可追溯

## 步骤

1. 运行质量检查
   - 运行 `make check`
   - 若失败，修复后继续，直到全部通过
   - 不要用抑制警告的方式掩盖问题（如 `#[allow(...)]`）

2. 查看发布范围
   - 找到最近标签：`git describe --tags --abbrev=0`
   - 审查变更：`git diff <previous-tag>..HEAD`

3. 处理未提交改动
   - 先看 `git status`
   - 只暂存发布相关文件，不要无差别提交全部改动
   - 如需提交，写清晰的提交信息

4. 更新 `CHANGELOG.md`
   - 新增当前版本章节
   - 用面向用户的中文描述变更
   - 重点写新增、修复、改进，避免纯技术噪音

5. 提交发布变更
   - 提交 `CHANGELOG.md` 和必要的版本文件
   - 使用 `chore: release <version>` 风格提交信息

6. 执行发布
   - 运行 `make release`
   - 若失败，停止并说明失败原因

7. 发布后反馈
   - 返回发布结果
   - 提供对应的 GitHub Actions 链接

## 执行原则

- 在质量检查通过前，不进入发布步骤
- 遇到不确定或高风险操作，先停下并询问用户
- 不做破坏性 Git 操作
